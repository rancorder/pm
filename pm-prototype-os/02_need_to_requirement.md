# 02 Need to Requirement

顧客の発言を、PMが扱える要件へ変換するためのルール。

## 変換原則

顧客の発言は、次の4層に分解する。

```text
発言
↓
要望
↓
課題
↓
要件
```

## 変換テンプレート

| 顧客発言 | 表面的な要望 | 本当の課題 | 要件 | 検証方法 |
|---|---|---|---|---|
|  |  |  |  |  |

## 要件分類

### Functional Requirement

システムが何をできるべきか。

- ファイルを読み込む
- 図面情報を抽出する
- モデルを検証する
- ルール違反を表示する
- レポートを出力する
- 自然言語で操作する

### Non-Functional Requirement

品質・制約・性能。

- 処理速度
- 精度
- セキュリティ
- 操作性
- 監査性
- 拡張性
- 既存環境との互換性

### Business Requirement

事業・業務成果。

- 工数削減
- 手戻り削減
- レビュー品質向上
- 属人化解消
- 顧客説明力向上
- 受注率向上

## 要件の良し悪し

悪い要件：

```text
AIで図面チェックできるようにする
```

良い要件：

```text
PDF図面またはIFCモデルを入力し、事前定義したチェックルールに対して不整合候補を抽出し、該当箇所・理由・確認優先度を一覧表示する
```

## PM判断ポイント

- 顧客が欲しいのは自動化か、判断支援か？
- 100%正解が必要か、候補提示でよいか？
- 入力データは構造化されているか？
- 現場で修正できるUIが必要か？
- 既存ツール内で完結すべきか？
- PoCで検証する最小単位は何か？
