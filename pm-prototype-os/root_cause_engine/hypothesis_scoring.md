# Hypothesis Scoring

真因候補を感覚ではなく、PM判断に使える形で優先順位づけする。

## スコアリングの目的

顧客の要望に対して、複数の真因候補を出し、どれから検証するか決める。

```text
真因候補を広げる
↓
証拠で重みづけする
↓
切り分け質問を作る
↓
MVPで検証する
```

## スコア項目

| 項目 | 意味 | 1点 | 5点 |
|---|---|---|---|
| Evidence | 顧客発言・資料の根拠 | 根拠薄い | 明確な根拠あり |
| Impact | 解けた時の業務効果 | 小さい | 大きい |
| Frequency | 発生頻度 | 稀 | 毎日/毎案件 |
| Urgency | すぐ困っているか | 後回し可 | 今すぐ必要 |
| Feasibility | 検証しやすさ | 難しい | すぐ検証可 |
| Data Readiness | データがあるか | ない | すぐ使える |
| MVP Fit | 小さく見せられるか | 見せにくい | 1画面で見せられる |
| Arent Fit | Arent領域と合うか | 遠い | 直結 |

## 優先度計算

```text
Priority Score = Evidence + Impact + Frequency + Urgency + Feasibility + Data Readiness + MVP Fit + Arent Fit
```

最大40点。

## 判定

| 点数 | 判定 | 対応 |
|---:|---|---|
| 32-40 | S | 最優先でMVP化 |
| 25-31 | A | 条件確認後にMVP化 |
| 18-24 | B | 保留。追加質問が必要 |
| 17以下 | C | 今回は扱わない |

## フロア制約（重要：単純加算の穴を防ぐ）

合計点だけで判定すると、根拠が薄い仮説でも他項目（特にArent FitやMVP Fit）が高ければSランクに届いてしまう。建設業のように証跡・責任が重い領域では、これは「もっともらしい数字に支えられた誤った確信」を生む危険な穴になる。

以下のフロア制約を、合計点の判定より先に適用する。

```text
if Evidence <= 2:
    判定は最大でもB（合計点に関わらず）

if Evidence <= 1:
    判定は最大でもC（合計点に関わらず）
```

理由：Evidenceは「顧客発言・資料の根拠」であり、ここが薄い仮説は、他がどれだけ魅力的でも検証する優先度を上げてはいけない。先にやるべきは検証ではなく、Evidenceを上げる質問。

## クイックスコア（実運用向け簡易版）

顧客打ち合わせ直後の30分ゴールでは、8項目すべてを毎回スコアリングするのは重い。仮説が3件を超える、または時間が厳しい場合は以下のコア4項目だけで一次判定し、MVP化を決めた仮説だけ後からフル8項目で精緻化する。

```text
Quick Score = Evidence + Impact + Feasibility + Arent Fit（最大20点）

15-20: 一次S候補（フルスコアで確定）
10-14: 一次A候補
9以下: 保留
```

フロア制約（Evidence<=2でB以下）はクイックスコアでも適用する。

## 切り分け質問の作り方

高スコア仮説同士を分ける質問を優先する。

例：

```text
探したいのは「ファイルとしての最新版」ですか？
それとも「図面内に書かれた条件や部材」ですか？
```

この1問で、版管理系とRAG系を切り分けられる。
