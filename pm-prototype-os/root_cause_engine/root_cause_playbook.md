# Root Cause Playbook

顧客要望から真因を追求し、MVPと実装案まで導くための実行手順。

## Step 0: 顧客要望を「症状」として扱う

顧客の発言は、最初は答えではなく症状として扱う。

例：

```text
顧客発言：図面をAIで検索したい
症状：必要な図面・情報にたどり着けていない
```

## Step 1: 要望を分解する

| 項目 | 内容 |
|---|---|
| 顧客発言 | そのままの言葉 |
| 表面的な要望 | 作ってほしいと言っているもの |
| 困っている作業 | 実際に時間がかかっていること |
| 困るタイミング | いつ発生するか |
| 影響 | 時間、品質、手戻り、コスト、判断遅れ |
| 関係者 | 誰が困っているか |

## Step 2: 真因候補を広げる

Fishbone的にカテゴリごとに仮説を出す。

| カテゴリ | 真因候補 |
|---|---|
| Data | データがない、古い、構造化されていない |
| Process | 手順・承認・変更管理が曖昧 |
| Tool | ツールが分断、検索性が低い |
| People | 属人化、教育不足、判断者不在 |
| Rule | 基準・ルール・命名規則がない |
| Organization | 部署間で責任分界が曖昧 |
| Contract | 契約範囲や追加対応が曖昧 |
| Cost | 費用判断・単価根拠が不明 |

## Step 3: 5 Whysで縦に掘る

各真因候補に対して「なぜ」を重ねる。

```text
なぜ困っている？
なぜその状態が起きる？
なぜ防げない？
なぜ検知できない？
なぜ標準化されていない？
```

ただし、5回にこだわらない。浅いと感じたら続ける。

## Step 4: Issue Treeで横に分解する

問題を原因系と解決系に分ける。

```text
問題：図面検索に時間がかかる

原因ツリー：
- データの問題
- ルールの問題
- ツールの問題
- 人の問題
- 業務フローの問題

解決ツリー：
- 検索改善
- 版管理改善
- 変更管理改善
- 教育支援
- BIM属性整備
```

## Step 5: 仮説をスコアリングする

| 仮説 | 確度 | 影響度 | 検証容易性 | MVP適合 | 優先度 |
|---|---:|---:|---:|---:|---:|
|  | 1-5 | 1-5 | 1-5 | 1-5 | 合計 |

### スコアの意味

- 確度：顧客発言や資料から見てありそうか
- 影響度：解けたときの効果が大きいか
- 検証容易性：短期間で検証できるか
- MVP適合：小さく作って価値を見せられるか

## Step 6: 切り分け質問を作る

良い質問は、仮説を絞る質問。

悪い質問：

```text
図面検索AIが欲しいですか？
```

良い質問：

```text
探せないのは、最新版が分からないからですか？
それとも図面内の情報が探せないからですか？
```

## Step 7: 真因を要件に変換する

| 真因 | 要件化 |
|---|---|
| 最新版が分からない | 図面版・更新日・承認状態を一覧管理する |
| 図面内情報が探せない | 図面テキスト・表題欄・関連仕様を検索可能にする |
| 議事録の変更が追えない | 議事録から変更指示を抽出し、図面反映状況を管理する |
| BIM属性が不足 | IFC属性欠損を検出し、用途別に不足項目を表示する |

## Step 8: MVPに切る

MVPは最小機能ではなく、真因仮説の検証単位。

```text
真因仮説
↓
検証したいこと
↓
最小入力データ
↓
最小画面
↓
成功条件
```

## Step 9: 実装案へ接続する

| MVPタイプ | 実装パターン |
|---|---|
| 資料検索 | RAG Evidence Viewer |
| 版管理 | Difference Review Table / Project Brain View |
| 変更管理 | Workflow Kanban / Decision Log |
| BIM属性 | File Analysis Dashboard |
| AI操作 | Natural Language Command Console |

## Step 10: PoC評価へ接続する

PoC後に必ず評価する。

```text
真因は正しかったか？
顧客は価値を感じたか？
入力データは使えたか？
現場フローに入るか？
本番化する価値はあるか？
```
