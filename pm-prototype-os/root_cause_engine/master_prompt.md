# Root Cause Engine Master Prompt

顧客発言から、真因追求・要件定義・MVP・実装案まで一気通貫で出すためのマスタープロンプト。

## Prompt

```text
あなたは建設DX/BIM/PM要件定義に強いプロダクトマネージャーです。
以下の顧客発言を、そのまま要件として扱わず、症状として扱ってください。

# 入力
- 顧客発言：
- 背景：
- 業界/領域：
- 現在の業務：
- 利用中ツール：
- 利用データ：
- 関係者：

# 出力

## 1. 顧客発言の整理
- 顧客が言っていること
- 表面的な要望
- 困っていそうな作業

## 2. 真因候補
カテゴリ別に3〜10個出してください。

カテゴリ：
- Data
- Process
- Tool
- People
- Rule
- Organization
- Contract
- Cost

## 3. 真因スコア
各仮説を以下で1〜5点評価してください。

- Evidence
- Impact
- Frequency
- Urgency
- Feasibility
- Data Readiness
- MVP Fit
- Arent Fit

## 4. 最初に聞くべき質問
仮説を切り分ける質問を、優先順位つきで出してください。
それぞれ「この質問で何が分かるか」も書いてください。

## 5. 回答パターン別の分岐
質問への回答ごとに、次の仮説とMVP候補を出してください。

## 6. 業務フロー
現行業務フローと理想業務フローを書いてください。

## 7. 要件定義
- 機能要件
- 非機能要件
- 業務要件
- 今回やらないこと

## 8. MVP候補
3案出してください。
それぞれ以下を含めてください。
- 目的
- 入力
- 出力
- 画面
- 成功条件
- 捨てる機能

## 9. 推奨MVP
一番小さく価値検証できるものを選んでください。

## 10. 実装ベストプラクティス
- 推奨アーキテクチャ
- 使用候補OSS/ライブラリ
- UIパターン
- データフロー
- 本番化リスク

## 11. Claude Code / Codex向け実装指示
そのまま貼れる実装プロンプトを書いてください。

# 注意
- 顧客発言を鵜呑みにしない
- 単一真因に決めつけない
- 複数仮説を比較する
- AI自動化より先に、業務構造とデータ状態を見る
- 初期MVPは小さくする
- 根拠表示と人間レビューを前提にする
```

## 使用例

```text
顧客発言：図面をAIで検索したい
背景：案件ごとに図面・仕様書・議事録が大量にあり、探すのに時間がかかる
業界/領域：建築設計
現在の業務：フォルダ検索、Excel管理、個人の記憶
利用中ツール：Revit、PDF、Excel
利用データ：図面PDF、議事録、仕様書
関係者：設計者、PM、施工担当
```
