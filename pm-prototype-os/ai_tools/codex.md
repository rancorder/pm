# Codex.md

PM Prototype OSにおけるCodexの役割。

## 役割

Codexは、プロトタイプ生成・小さな実装・テスト・PR単位作業の担当。

主な用途は以下。

- Webプロトタイプ作成
- Next.js / React UI作成
- Pythonスクリプト作成
- APIモック作成
- ダミーデータ生成
- テスト作成
- 小規模機能追加
- PR作成前の実装整理

## 得意な仕事

```text
PM要件
↓
実装タスク
↓
コード生成
↓
動くプロトタイプ
↓
テスト/修正
```

## 使うタイミング

- 顧客確認用の画面をすぐ作りたいとき
- 要件を動く形にしたいとき
- UIモックだけでは弱いとき
- ダミーデータで業務フローを見せたいとき
- 小さな検証コードを書きたいとき
- テストやサンプルを作りたいとき

## 標準プロンプト：プロトタイプ生成

```text
以下のPM要件から、顧客確認用のWebプロトタイプを作成してください。

# 目的
顧客が業務改善イメージを理解し、次回打ち合わせで要件差分を議論できる状態にする。

# 制約
- 本番実装ではなくプロトタイプ
- ログイン不要
- DB不要。必要ならダミーデータでよい
- UIは業務システムらしく簡潔に
- 画面上でBefore/Afterが伝わること
- 顧客に説明しやすいこと

# 出力
1. 実装方針
2. ファイル構成
3. 実装コード
4. ダミーデータ
5. 起動方法
6. 顧客説明ポイント
7. 本番化する場合の追加課題
```

## 標準プロンプト：小規模機能実装

```text
以下の機能を既存プロジェクトに追加してください。

# 要件

# 制約
- 変更範囲を小さくする
- 既存コードの設計に合わせる
- 不要なライブラリ追加を避ける
- テスト可能な形にする

# 出力
- 変更ファイル
- 実装内容
- テスト方法
- 残課題
```

## Codexにやらせないこと

- 要件が曖昧なまま大規模実装
- 本番DBや本番APIへの接続
- 秘密情報の読み書き
- ライセンス未確認コードの取り込み
- 顧客データをダミー化せずに利用
- プロトタイプを本番品質として扱うこと

## PMとしての確認ポイント

- 顧客に見せる価値があるか
- 1画面で業務改善が伝わるか
- ダミーデータでも検証できるか
- 実装が大きくなりすぎていないか
- 次回顧客確認につながるか
- 本番化リスクが明記されているか
