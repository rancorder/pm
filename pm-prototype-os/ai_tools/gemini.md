# Gemini.md

PM Prototype OSにおけるGeminiの役割。

## 役割

Geminiは、マルチモーダル一次解析の担当。GPT/Claudeがテキストに変換した後の処理を受け持つのに対し、Geminiは「テキスト化される前」の生データの入口を受け持つ。

主な用途は以下。

- スキャンPDF図面・手書きメモの一次読解
- 音声議事録の文字起こし＋構造化（既存の動画文字起こしパイプラインの延長）
- 長尺の打ち合わせ録音から、決定事項・未決事項・宿題を直接抽出
- 写真（現場写真・ホワイトボード撮影）からのテキスト/図面情報抽出

## 得意な仕事

```text
生データ（音声/画像/スキャンPDF）
↓
マルチモーダル理解
↓
構造化テキスト
↓
Claude/GPTに渡せる形
```

## 使うタイミング

- 議事録が音声/動画のまま手元にあるとき
- 図面がスキャンPDFで、テキスト抽出ツールが効かないとき
- 現場写真やホワイトボードの内容を記録に残したいとき
- 既存の動画文字起こしパイプライン（Gemini API + Drive API）を流用できる案件のとき

## 標準プロンプト：音声議事録の構造化

```text
以下の打ち合わせ音声/文字起こしから、PMが要件定義に使える形で構造化してください。

# 出力
1. 会議の目的
2. 発言者ごとの要約
3. 決定事項
4. 未決事項（担当者・期限があれば含める）
5. 顧客の感情・温度感（急いでいる/様子見など）
6. 表面的な要望
7. 次回確認すべきこと

# 注意
- 発言と推測を分ける
- 聞き取れなかった箇所は明示する
- 専門用語は可能な範囲で正規化する（表記ゆれを統一する）
```

## 標準プロンプト：スキャン図面の一次読解

```text
以下のスキャン図面/手書きメモ画像から、テキストとして読み取れる情報を抽出してください。

# 出力
- 表題欄の情報（図面番号、版、作成日、案件名）
- 記載されている寸法・部材名・仕様の候補
- 読み取れなかった/不確実な箇所
- 後続のRAG/検索に使えそうなキーワード

# 注意
- 読み取れない箇所は「不明」と書き、推測で埋めない
- OCR的な読み取りであり、設計判断は行わない
```

## Geminiにやらせないこと

- 設計判断・要件の確定（一次解析の担当であり、PM判断はGPT/Claudeに渡す）
- 読み取れない箇所の推測による埋め合わせ
- 顧客の機密音声・画像を許可なく外部に保持させる運用

## PMとしての確認ポイント

- 一次解析結果が、後続のClaude（議事録整理）・GPT（要件定義）にそのまま渡せる形になっているか
- 「不明」「読み取れない」が適切に残されているか（自信過剰な断定になっていないか）
- 同じ情報をGeminiとOCRツール（PyMuPDF/Tesseract等）の両方で扱う必要があるかの判断ができているか

## 既存パイプラインとの接続

動画文字起こし自動パイプライン（Gemini API + Drive API、Pydantic構造化出力）をすでに運用中であれば、ここでの「音声議事録の構造化」プロンプトはそのパイプラインの出力フォーマットに合わせて調整する。PM Prototype OS側のテンプレートとパイプライン側のPydanticスキーマで項目名がズレないように、定期的に突き合わせること。
