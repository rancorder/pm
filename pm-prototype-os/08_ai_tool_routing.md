# 08 AI Tool Routing

PM Prototype OSで、GPT / Claude / Claude Code / Codexをどう使い分けるかの司令塔。

## 基本方針

AIツールは同じ仕事をさせない。役割を分ける。

```text
GPT
→ 構造化・戦略・要件定義・論点整理

Claude
→ 長文資料読解・議事録整理・仕様書化・文章生成

Claude Code
→ 既存リポジトリ理解・修正・リファクタ・実装補助

Codex
→ プロトタイプ生成・コード実装・テスト・PR単位の作業
```

## PM業務フローでの使い分け

| 工程 | 主担当AI | 補助AI | 出力 |
|---|---|---|---|
| 顧客ヒアリング後の整理 | GPT | Claude | 課題構造・論点 |
| 議事録の清書 | Claude | GPT | 議事録・要約 |
| 要件定義 | GPT | Claude | 要件一覧・MVP |
| 長文資料読解 | Claude | GPT | 仕様理解・差分 |
| GitHub repo理解 | Claude Code | Codex | 構成理解・実装方針 |
| プロトタイプ生成 | Codex | Claude Code | 動くPoC |
| 既存コード改修 | Claude Code | Codex | 修正PR |
| レビュー観点整理 | GPT | Claude | PMレビュー観点 |
| エンジニア依頼文 | Claude | GPT | 依頼文・仕様共有 |

## 使い分けの判断基準

### GPTを使う場面

- 抽象度が高い
- 意思決定が必要
- 課題の構造化が必要
- 営業・PM・事業視点が混ざる
- 「何を作るべきか」を決めたい

### Claudeを使う場面

- 長文を読む
- 議事録をまとめる
- 資料の抜け漏れを見る
- 仕様書・説明文をきれいに書く
- 顧客向け文章を作る

### Claude Codeを使う場面

- 既存コードを読ませたい
- リポジトリ全体を理解したい
- どこを直すべきか調べたい
- 実装方針をファイル単位で出したい
- エラー原因を追いたい

### Codexを使う場面

- 新規プロトタイプを作る
- 小さな機能を実装する
- テストを書く
- PR単位で作業させる
- Next.js / Python / APIなどを一気に形にする

## 禁止事項

- 顧客資料をそのまま外部公開repoに入れない
- 秘密情報を公開AIに貼らない
- OSSコードをライセンス確認なしに取り込まない
- AIに危険なコマンドを無確認で実行させない
- プロトタイプを本番品質と誤認しない

## 最強パターン

```text
1. Claudeで議事録と資料を整理
2. GPTで課題・要件・MVPを構造化
3. Claude Codeで既存repo/技術候補を読む
4. Codexでプロトタイプを作る
5. GPTで顧客説明・次回質問を作る
```
