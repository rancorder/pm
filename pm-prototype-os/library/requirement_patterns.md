# Requirement Patterns

建設DX/PM業務で繰り返し使う要件定義パターン集。

## 基本フレーム

すべての要件は次の形に落とす。

```text
入力
↓
処理/判断
↓
出力
↓
確認/承認
↓
証跡
```

## Pattern 01: Upload → Analyze → Review → Report

### 使う場面

- PDF図面解析
- 仕様書チェック
- 見積比較
- 写真分類
- IFCチェック

### 要件テンプレート

```text
ユーザーがファイルをアップロードすると、システムが内容を解析し、確認候補を一覧表示する。ユーザーは候補を確認・修正し、最終的にレポートを出力できる。
```

### 必要画面

- ファイルアップロード
- 解析結果一覧
- 詳細確認
- レポート出力

### 注意点

- AIの結果は確定ではなく候補として扱う
- 根拠表示が必須
- 元ファイルへの参照リンクを持つ

---

## Pattern 02: Search → Evidence → Action

### 使う場面

- 図面/仕様書RAG
- 議事録検索
- 設計ルール検索
- 点検履歴検索

### 要件テンプレート

```text
ユーザーが質問を入力すると、関連資料から回答候補と根拠を提示し、必要な次アクションを表示する。
```

### 必要画面

- 検索/質問入力
- 回答
- 根拠資料
- 次アクション

### 注意点

- 根拠のない回答は禁止
- 版管理を考慮する
- 回答できない場合は不足資料を示す

---

## Pattern 03: Rule Check → Violation Candidate → Human Approval

### 使う場面

- BIM属性チェック
- IDS検証
- 図面ルールチェック
- 配筋ルール確認

### 要件テンプレート

```text
事前定義したルールに対して対象データをチェックし、違反候補を表示する。最終判断は人間が行い、承認・却下・保留を記録する。
```

### 必要画面

- ルール一覧
- チェック実行
- 違反候補一覧
- 承認/却下
- 履歴

### 注意点

- ルールの定義元を明示する
- AI判断とルールベース判断を分ける
- 最終責任者を明確にする

---

## Pattern 04: Meeting Minutes → Decision Log → Change Tracking

### 使う場面

- 議事録から設計変更抽出
- 未決事項管理
- 顧客要望管理
- 課題管理

### 要件テンプレート

```text
議事録から決定事項・未決事項・変更指示・宿題を抽出し、担当者・期限・影響対象を管理する。
```

### 必要画面

- 議事録投入
- 抽出結果
- 変更ログ
- 担当/期限管理
- 未反映一覧

### 注意点

- 発言者と決定者を分ける
- 決定事項と要望を混ぜない
- 図面/仕様/工程への反映状況を持つ

---

## Pattern 05: Comparison Table → Difference → Clarification Question

### 使う場面

- 見積比較
- 図面版差分
- 仕様書差分
- OSS比較
- 技術選定

### 要件テンプレート

```text
複数資料を比較し、差分・不足・確認質問を一覧表示する。
```

### 必要画面

- 比較対象選択
- 比較表
- 差分一覧
- 確認質問

### 注意点

- 単純な金額差だけで判断しない
- 条件差を必ず表示する
- 比較軸をユーザーが編集できると強い

---

## Pattern 06: Workflow Status → Bottleneck → Next Action

### 使う場面

- 工程管理
- 是正管理
- 検査進捗
- 承認フロー

### 要件テンプレート

```text
各タスクの状態を可視化し、遅延・未対応・ボトルネックを提示し、次アクションを提案する。
```

### 必要画面

- ステータス一覧
- ボトルネック表示
- 担当者別タスク
- 次アクション

### 注意点

- 現場入力負荷を増やさない
- 通知疲れを防ぐ
- 本社KPIと現場実態を分ける

---

## Pattern 07: Natural Language → Operation Plan → Safe Execution

### 使う場面

- Revit操作AI Agent
- BIM操作支援
- 自動レポート生成
- ファイル操作自動化

### 要件テンプレート

```text
ユーザーが自然言語で指示すると、システムは実行前に操作計画・影響範囲・リスクを表示し、ユーザー承認後に実行する。
```

### 必要画面

- 自然言語入力
- 意図分類
- 操作計画
- 確認/承認
- 実行ログ

### 注意点

- いきなり実行しない
- 破壊的操作は必ず確認
- 操作ログを残す

---

## Pattern 08: Asset → History → Recommendation

### 使う場面

- 設備台帳
- 維持管理
- 点検履歴
- 修繕判断

### 要件テンプレート

```text
対象資産の履歴を集約し、過去の点検・修繕・異常から次の対応候補を提示する。
```

### 必要画面

- 資産一覧
- 履歴タイムライン
- 関連資料
- 推奨対応

### 注意点

- 推奨は判断支援にとどめる
- 法定点検と任意点検を分ける
- 履歴の欠損を表示する

---

## Pattern 09: Past Case → Similarity → Reuse

### 使う場面

- 過去案件検索
- 設計判断再利用
- 見積単価参照
- トラブル事例検索

### 要件テンプレート

```text
現在の案件条件から類似する過去案件を検索し、再利用可能な判断・資料・注意点を提示する。
```

### 必要画面

- 案件条件入力
- 類似案件一覧
- 再利用可能情報
- 注意点

### 注意点

- 類似理由を表示する
- 条件違いを明示する
- 古い情報をそのまま使わない

---

## Pattern 10: KPI → Drilldown → Root Cause

### 使う場面

- 工程遅延分析
- 品質不良分析
- コスト超過分析
- PMダッシュボード

### 要件テンプレート

```text
KPIの異常を検知し、関連するデータへ掘り下げ、原因候補と確認先を提示する。
```

### 必要画面

- KPIダッシュボード
- 異常一覧
- ドリルダウン
- 原因候補
- 確認先

### 注意点

- 数字だけで断定しない
- 現場コメントを併記する
- 原因候補と確定原因を分ける
