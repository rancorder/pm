# PoC Evaluation

PM Prototype OSで作ったプロトタイプを、次フェーズへ進めるか判断するための評価基準。

## PoCの目的

PoCは「作れるか」ではなく「進める価値があるか」を判断するために行う。

```text
技術検証
＋
顧客価値検証
＋
業務適合検証
＋
事業性検証
```

## 評価軸

| 評価軸 | 見ること | 判定 |
|---|---|---|
| 顧客価値 | 顧客が本当に使いたいか | S/A/B/C |
| 業務適合 | 現場フローに入るか | S/A/B/C |
| 技術実現性 | 実装可能か | S/A/B/C |
| データ適合 | 入力データが使えるか | S/A/B/C |
| 精度許容 | AI出力の精度が許容範囲か | S/A/B/C |
| 運用負荷 | 入力・確認・管理が重すぎないか | S/A/B/C |
| 本番化リスク | セキュリティ・権限・責任が許容可能か | S/A/B/C |
| 事業性 | 予算・ROI・横展開性があるか | S/A/B/C |

## S/A/B/C 判定

### S: すぐ進める

- 顧客が明確に欲しいと言った
- 実データで価値が出た
- 本番化の道筋が見える
- 横展開できる

### A: 条件付きで進める

- 価値はあるが、データや運用に条件がある
- 追加検証が必要
- 顧客側の意思決定者確認が必要

### B: 保留

- 面白いが業務導入が弱い
- 技術的にはできるが費用対効果が不明
- 顧客の課題優先度が低い

### C: やらない

- 顧客価値が弱い
- データが使えない
- 責任リスクが高い
- 既存ツールで十分

## PoCレビュー質問

### 顧客価値

- 顧客はどの画面で一番反応したか？
- 「これ欲しい」と言ったか？
- 誰の何時間が減るか？
- 顧客はお金を払う理由を説明できるか？

### 業務適合

- 誰が入力するか？
- 誰が確認するか？
- 誰が承認するか？
- 既存業務のどこに入るか？
- 現場で本当に使えるか？

### 技術

- PoCと本番で設計が大きく変わるか？
- データ量が増えても耐えられるか？
- 精度はどこまで必要か？
- 人間レビューで許容できるか？

### 事業性

- 他社にも売れるか？
- Arent既存プロダクトと接続できるか？
- 継続利用されるか？
- 単発開発で終わらないか？


## PM Brainへの反映（重要）

この評価結果は、その場で消費して終わりにしない。`pm_brain/cases/`の該当案件ファイルに、以下の対応でそのまま転記する。

| このファイルの結論 | `pm_brain`側のfrontmatter `poc_result` |
|---|---|
| Go | `Go` |
| Conditional Go | `Conditional Go` |
| Hold | `Hold` |
| No-Go | `No-Go` |

「次に検証すること」「顧客に確認すること」の内容は、案件ファイルの「次案件に使える学び」セクションに要約して残す。これを次回似た案件で`search_cases.py`から引けるようにすることが、PM Brainが機能する（＝案件を重ねるほど賢くなる）ための最低条件。

## 典型的な失敗判定

- デモは受けたが、利用者がいない
- 顧客は面白いと言ったが、予算責任者がいない
- AI精度より、データ準備がボトルネックだった
- 自動化対象の業務が標準化されていなかった
- 本番化した瞬間に権限・監査・責任問題が出た
