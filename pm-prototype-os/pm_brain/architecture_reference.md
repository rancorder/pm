# PM Brain Architecture Reference

`pm_brain/`の設計（v0.2で最小実装済み）は、公開されているPM向けメモリシステムOSSの設計思想を参考にしている。ここではそのOSSのコードや文章を引用せず、**採用した設計思想だけを一般化して**記録する（参照: https://github.com/phuryn/pm-brain）。

## 参考にした3つの設計判断

### 1. 決定論的スキャフォールド＋適応的プロンプトの分離

案件データを保存する「型」（frontmatter・見出し構成）はプログラムで固定し、その型の中身をどう埋めるかはAI（適応的プロンプト）に任せる、という役割分担。型自体をAIに毎回考えさせると、案件ごとにフォーマットがブレて検索できなくなる。

→ PM Prototype OSでの反映：`pm_brain/cases/_template.md`のfrontmatterキー（`industry` / `phase` / `cause_category`等）は固定スキーマとして扱い、`search_cases.py`はこのスキーマに依存する。AIに案件を書かせるときも、このキー名は変えさせない。

### 2. Migrationは「今動いている案件」だけを対象にする

過去の資料を遡及的に大量投入しない。今進行中の案件・直近の意思決定だけを取り込む。理由は、古い資料を一気に入れると初期のノイズで検索精度が落ち、かつ「今のプロジェクトにとって重要な情報」と「昔は重要だったが今は違う情報」の区別がつかなくなるため。

→ PM Prototype OSでの反映：`pm_brain/README.md`の運用ルールに明記済み。sales-os等で蓄積した過去のナレッジをそのままpm_brainに輸入しない。

### 3. 評価（eval）を伴わないメモリシステムは信用できない

「動いているように見える」だけでは不十分で、実際に正しい情報を思い出せているかを検証するテストスイートを持つ、という思想。

→ PM Prototype OSでの反映：現状`search_cases.py`は動作確認レベル（デモケース1件での手動確認）にとどまっている。案件数が10件を超えた時点で、`root_cause_engine/hypothesis_scoring.md`の判定結果と実際のPoC結果（`library/poc_evaluation.md`のGo/No-Go）を突き合わせる簡易評価を導入することを推奨する（次のイテレーションのタスク）。

## あえて採用しなかったもの

- ベクトルDB・埋め込み検索：案件数が数十件規模になるまでは、frontmatterベースの構造化検索で十分機能する。過剰実装を避けるという本OS自体の思想（`failure_patterns.md`の「08 技術選定が目的化する」）とも一致する
- グローバルスキル化（`~/.claude/skills/`的な、全プロジェクト横断のインストール形式）：PM Prototype OSは特定の案件文脈（Arent/建設DX）に強く紐づいているため、汎用スキルとして外出しするメリットが薄い

## 次に案件が10件を超えたら見直すこと

- frontmatterのキーが実際の検索ニーズに対して過不足ないか（使っていないキー、逆に欲しいのに無いキーがないか）
- `search_cases.py`のキーワード検索がgrep相当で十分か、簡易的な類義語対応（例：「版管理」で検索して「バージョン管理」の案件も拾えるか）が必要か
